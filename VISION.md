# VISION: Immutable Package Store

本文档用于在第一个 RFC 之前讨论项目图景。它不是决策记录，也不是实现方案；它的作用是把我们对项目的直觉、目标、边界和待澄清问题先放到同一个地方，等图景足够稳定后，再沉淀为正式 RFC。

当前讨论重点是三条主线：

- 从外部视角看的使用体验：`mise`、普通用户和管理员如何感知 `imst`，以及按需安装、状态观察、复用结果在体验上应是什么样。
- 从能力视角看的概念模型：`imst` 提供哪些基础能力，如何理解请求、对象、store path、全量意图、保活和状态。
- 从安全视角看的边界和控制：普通用户、`mise`、`imst daemon`、共享 store 之间的权限边界、只读保障、注册关系和 GC 控制。

本文档暂不深入具体实现细节，例如文件写入原子性、watch 机制、锁、具体系统调用或内部数据结构。

## 1. 一句话图景

`imst` 暂定来自 immutable package store，目标是在多用户环境下，为 `mise` 这样的用户态工具版本管理器提供一个底层的、共享的、不可变的包存储机制。

它不是包管理器，而是包管理器之下的一层基础设施：

- 面向多用户系统中的共享存储，而不是单个用户私有目录。
- 面向 `mise` 这类用户态工具版本管理器，而不是系统级发行版包管理器。
- 负责把远端或本地包源安全地变成 store 中的不可变内容。
- 不负责决定用户环境里应该激活什么包。

这里的关键不是“重新实现 `mise`”，而是借由 `mise` 这个代表性上层工具，讨论用户态包管理器对底层共享 store 的诉求。版本选择、工具激活、环境变量管理、命令暴露、用户 profile 管理等能力仍然属于 `mise` 或其他上层工具。

## 1.1 名称

`imst` 这个名字还没有完全确定。当前可理解为 `immutable package store` 的缩写。

待确认：

- 名称是否最终采用 `imst`。
- `imst` 是否读作一个词，还是按字母读。
- 项目是否强调 `immutable`，还是更强调 `package store`。
- 是否需要避免和已有项目、命令或包名冲突。

## 2. 为什么要做

现有用户态包管理器的一个共性问题是：同一台机器上的大量重复包会被反复下载、缓存、解压和存储，占据非常多的磁盘空间。

这类问题在用户态生态中很常见。比如 Rust/Cargo、Go、Zig 以及其他语言或工具链的包管理机制，通常都会围绕当前用户、当前项目或当前工具维护自己的缓存和包目录。`mise` 这类工具版本管理器还会进一步安装多种语言运行时和开发工具。它们各自的设计在单用户、单生态、单工具视角下是合理的，但放到多用户机器或长期使用的开发环境中，就会自然积累出大量内容相同或高度重复的包副本。

这种重复来自几个层面：

- 不同用户会各自拥有一份相同语言生态的包缓存。
- 不同项目会各自保留相同依赖的副本或构建输入。
- 不同工具链之间缺少可共享的底层存储抽象，即使内容相同，也难以复用。
- 包管理器通常把“包选择、环境管理”和“包内容存储”耦合在一起，使得底层存储很难跨用户、跨工具复用。

`imst` 的动机是先把这个问题作为独立问题看待：在不取代上层包管理器、不接管用户环境的前提下，是否可以提供一个共享的、不可变的底层包存储，让重复的工具版本和包内容只需要在机器上稳定保存一份。`mise` 在当前讨论中主要作为思想实验和代表性实体，用来帮助收敛 `imst` 的外部诉求。

这个目标的难点不只是“把文件放到同一个目录”。真正的矛盾在于：使用方，也就是 `mise` 这类包管理器或同一安全边界内的用户意图来源，希望保持用户态的按需安装体验，并且在拿到 store path 后获得只读保障；但供应方，也就是负责向共享 store 写入和发布内容的一侧，又天然需要某种特权来完成跨用户共享写入、权限收敛和只读发布。

换句话说，消费者侧诉求是“普通用户可以按需请求安装，并且得到不会被其他普通用户篡改的只读内容”；供应侧诉求是“有能力把内容写入多用户共享位置，并在发布后收回写权限”。这两个诉求之间存在结构性张力。如果 store object 不能在写入完成后可靠地变成只读、不可变的内容，共享 store 反而会引入污染和越权风险；但如果所有写入都要求用户直接拥有共享 store 的写权限，又会破坏只读保障和权限边界。

第二个难点是保活和 GC。共享 store 的价值来自复用，但复用也意味着 store object 的生命周期不再只属于某一个用户或某一个包管理器实例。`mise` 可以在用户环境中引用某个 store path，但 `imst` 如果不知道这个引用是否仍然有效，就无法安全地判断对象是否可以删除。过早 GC 会破坏仍在使用的用户环境；过于保守则会让共享 store 重新变成只增不减的缓存堆积。

因此，`imst` 需要讨论的不只是“如何删除不用的对象”，而是“谁有资格声明对象仍然需要保活、声明的粒度是什么、声明如何过期或撤销、GC 如何在多用户环境中避免误删”。这同样是底层 store 和上层包管理器之间的边界问题：`imst` 能管理对象生命周期，但它不能擅自理解或修改 `mise` 的用户环境。

因此，项目首先关心的不是如何设计一个新的包管理器，也不是如何真实改造 `mise`，而是如何减少用户态工具版本和包内容在多用户环境中的重复存储成本，并为上层包管理器或工具链提供一个可以共同依赖的底层存储机制。

## 3. 服务对象

`mise` 是当前讨论中的代表性服务对象。更准确地说，`imst` 服务的是上层工具背后的下载、校验、展开和存储诉求，而不是终端用户的 shell 环境。

主要使用者包括：

- `mise`：把工具版本的包源交给 `imst` 获取、校验和展开，再基于返回的 store 路径自行管理激活逻辑。
- 多用户机器的管理员或平台维护者：配置共享 store 的位置、权限、清理策略和可信策略。
- 未来的用户态包管理器、构建或开发工具：在需要获取外部包内容时，把 `imst` 作为统一的本地不可变 store。

终端用户可能间接受益于更快的工具安装、更少的磁盘占用和更稳定的工具内容，但普通用户不需要直接感知 `imst` 的存在。`imst` 是 `mise` 背后的存储机制，而不是面向普通用户的交互入口。

从代表性上层工具的外部体验看，`imst` 不是完全透明的系统级拦截层。类似 `mise` 的工具需要被配置为使用 `imst` 作为底层 store 能力。这个方向意味着未来大概率需要对具体上层工具做适配或 PR，但当前讨论重点不是如何改造 `mise`，而是让 `imst` 的模型足够清晰，使这类工具可以被改造成能够利用它。

这个体验成立的前提是：当前用户已经和 `imst` 建立了信任连接。这个连接可以通过用户加入 `imst-user` 组来表达。加入该组以后，用户获得的不是直接写共享 store 的权限，而是被 `imst daemon` 识别为可以声明个人全量意图的消费者。

因此，常规路径里，普通用户仍然使用上层包管理器的命令和配置；`imst` 的 CLI 或管理入口主要面向管理员、调试和系统维护，而不是日常工具安装体验。

当某个工具版本尚未就绪时，上层包管理器是选择等待、返回 pending、提示用户重试，还是采用其他交互方式，属于具体包管理器的适配策略。`imst` 只需要表达请求状态和 store object 是否就绪，不规定上层包管理器面向用户的等待或提示体验。

## 4. 核心场景

第一批核心场景应围绕“多个用户通过 `mise` 共享同一份已校验、已展开的工具版本内容”展开。

候选场景：

- 用户 A 通过 `mise` 安装某个工具版本，例如某个版本的 Node.js、Python、Go、Rust 或 Zig。`mise` 请求 `imst` 下载、校验并展开该工具版本。`imst` 把结果写入共享 store，并返回不可变 store 路径。
- 用户 B 之后也通过 `mise` 请求相同工具版本。`imst` 发现 store 中已经存在已校验对象，直接返回已有路径，不重复下载和展开。
- `mise` 拿到 store 路径后，自行决定如何在用户环境中激活该工具版本。`imst` 不参与环境变量、shim、symlink 或 shell 集成。

成功标准不是“用户能直接通过 `imst` 运行某个工具”，而是“`mise` 能稳定获得一份可验证、不可变、可共享的工具版本内容”。

暂不把通用构建缓存、远程制品仓库或部署系统作为第一核心场景，除非后续讨论确认它们是服务 `mise` 的自然延伸。

## 5. 产品形态

初始产品形态更接近本地系统组件，而不是完整应用：

- 作为上层包管理器背后的本地 store 能力存在。
- 对普通用户隐藏在上层包管理器之后。
- 对管理员提供必要的配置、诊断和维护入口。
- 对上层工具提供表达意图、观察状态和获取 store path 的能力。

不倾向于一开始做 Web 应用、公共 SaaS 或面向终端用户的图形界面。

## 6. 核心概念

待定。

项目里可能长期存在的领域对象、关系和状态应该在这里逐步浮现。第一版不需要完整，但要避免只讨论页面或接口。

可以先从这些问题开始：

- 系统处理的主要对象是什么？
- 对象之间有什么关系？
- 哪些状态变化最重要？
- 哪些数据需要长期保存？
- 哪些数据只是临时计算结果？

围绕 immutable package store，可能出现的核心概念包括：

- Package Source：上层包管理器提供给 `imst` 的包来源，可以是 URL、本地文件、归档文件或其他可获取来源。
- Package Spec：用户全量意图中描述一个包的声明，至少包含 `name` 和 `items`。
- Name：来自 `spec.name`，用于表达上层意图中的包名。
- Rev：来自 `hash(spec)` 的确定性计算结果，用于表达该 package spec 的修订。该算法必须是公开知识，使用方可以自行计算并利用它。
- Verification：包内容进入 store 前必须经过的校验过程，例如哈希校验、签名校验或上层提供的完整性约束。
- Capability Set：`requests.json` 能表达的有限能力集合，例如获取、校验、展开或放置文件；它不应包含执行脚本这类开放性能力。
- Immutable Source：`src` 支持的源头需要满足 immutable 假设，也就是同一地址应获得确定性内容。这个前提并不总是能由 `imst` 完全校验。
- Store Object：store 中不可变的底层对象，通常对应一个 package spec 的安装结果。
- Installed：某个 package spec 经过确定性处理后的安装结果。
- Store Path：`imst` 返回给 `mise` 或其他上层工具的不可变路径。
- Identity：对象身份，可能来自内容哈希，也可能组合来源、校验信息、展开方式和有限初始化参数。
- Metadata：用于描述来源、校验、展开方式、创建时间、所有权或引用关系的数据。
- Initialization：包展开后的有限初始化行为，只能作用于 store object 本身，不能修改用户环境。
- Permission Convergence：store object 从创建、写入、校验到发布过程中，权限逐步收敛到可共享、不可被普通用户修改的状态。
- Read-only Control：store object 发布后应对使用者表现为只读内容，避免后续安装或使用过程污染共享对象。
- Supplier Privilege：负责写入和发布共享内容的一侧可能需要高于普通使用者的权限，用于完成跨用户共享写入和权限收敛。
- Consumer Demand：使用方希望保持用户态按需安装体验，同时获得只读、可信、可复用的 store path。
- Liveness：某个 store object 当前仍被用户、配置、工具版本选择或上层状态需要保留。
- Root Set：使 store object 保活的显式声明集合，由所有已建立信任连接用户的 `requests.json` 共同构成，也可能叠加管理员策略。
- Garbage Collection：清理不再被任何用户或上层管理器引用的 store object。

## 7. 设计原则

待定。

这里记录项目的偏好和取舍，例如：

- 本地优先还是云端优先？
- 简单配置还是强可扩展？
- 强一致性还是最终一致性？
- 面向个人效率还是团队协作？
- 优先脚本化、可组合，还是优先图形界面体验？

与当前方向相关的候选原则：

- 不可变优先：store 中已确认的对象不应被原地修改。
- 可验证优先：对象身份应能被校验，而不是只依赖外部命名。
- 纯函数模型优先：安装过程在概念上应可理解为 `fn(spec) -> Installed`。给定同一个 spec，应得到同一个 installed 身份和内容。
- 有限能力优先：用户全量意图只能表达受限 store 能力，不能表达任意代码执行、post-install script 或开放式系统操作。
- immutable source 优先：`src` 代表的来源应满足同一地址获得确定性内容的假设；`imst` 可以校验获得的内容是否匹配 digest，但不一定能证明源头本身长期 immutable。
- 多用户共享优先：相同内容应能在多个用户、多个上层工具之间复用。
- 权限收敛优先：共享 store 中的对象不能长期停留在创建者可随意修改、其他用户又会复用的状态。
- 只读发布优先：store object 只有在写入、校验和必要初始化完成后，才应作为只读对象发布给 `mise` 使用。
- 特权隔离优先：需要特权的是供应和发布共享内容的过程，而不是 `mise` 的环境激活或普通用户的日常使用过程。
- 单一供应方优先：共享 store 中的 installed 内容只能由 `imst daemon` 发布；普通用户和上层工具只表达意图，不直接写入安装结果。
- 显式保活优先：GC 不能依赖猜测用户环境状态，应尽量依赖上层工具或管理策略提供的显式保活信息。
- 谨慎 GC 优先：在引用状态不明确时，误删仍被使用的对象比暂时保留无用对象更严重。
- 职责边界优先：`imst` 只负责底层存储，不替包管理器做环境管理。
- 非侵入优先：`imst` 不能修改用户环境变量、不能挂载 command shim、不能创建面向用户环境的 symlink。
- 可组合优先：项目首先应能被 `mise` 调用，同时不排除未来被其他用户态包管理器、脚本或工具调用。
- 本地优先：第一阶段先解决本机共享 store，再讨论远程同步或分发。

## 8. 明确不做

这些边界是项目定位的一部分，不只是第一阶段的裁剪：

- `imst` 不是包管理器。
- `imst` 不负责依赖解析和版本求解。
- `imst` 不负责管理依赖关系或解释依赖图。
- `imst` 不负责决定用户安装哪些包。
- `imst` 不负责修改用户 shell 配置、环境变量或 profile。
- `imst` 不挂载 command shim。
- `imst` 不在用户环境中创建用于暴露命令的 symlink。
- `imst` 不侵入用户 home 目录中的运行环境状态。

候选非目标：

- 第一阶段不急于提供公共 SaaS 制品仓库。
- 第一阶段不急于覆盖所有语言生态。
- 第一阶段不急于支持复杂构建过程；如果存在初始化行为，也应是有限的、可控的、只作用于 store 内容本身。
- 第一阶段不把“共享目录可写”视为可接受的共享机制；如果对象会被多个用户信任和复用，就必须讨论发布后的只读约束。
- `requests.json` 不应包含执行脚本、任意命令、post-install hook 这类开放性能力。

## 9. 第一阶段成功标准

第一阶段应验证最小闭环：

- `mise` 或模拟 `mise` 的测试驱动程序可以向 `imst` 提交工具版本来源和校验信息。
- `imst` 能下载或读取该来源。
- `imst` 能校验内容。
- `imst` 能把内容展开到共享 store 中的不可变位置。
- 多个用户或多次请求相同工具版本内容时，可以复用同一个 store object。
- `imst` 返回 store path，但不修改任何用户环境。

如果这个闭环成立，再进一步讨论锁、权限模型、GC、远程来源策略、初始化能力和与 `mise` 的接口。

第一阶段范围应限定在预编译分发物：

- 支持 tarball、zip、单文件 binary、普通文件这类已经存在的分发物。
- 负责获取、校验、展开或放置，并发布为只读 store object。
- 不做源码构建。
- 不执行任意 install script。
- 不支持 post-install hook。
- 不做依赖解析。
- 如果存在初始化，也只能是有限、声明式、可预测的处理。

## 10. 概念性设计图景

本节记录当前的概念模型，用于后续 RFC 展开。它刻意不固定具体路径、文件格式、进程通信方式或系统调用。

### 10.1 角色与边界

- 消费者：普通用户或其安全边界内的上层工具，只能表达个人全量意图。
- 供应方：`imst daemon`，负责根据全量意图获取、校验、处理并发布 store object。
- 管理者：配置共享 store、用户接入、策略和维护行为的角色。

消费者不是管理者。消费者不能直接写共享 store，也不能发布 installed object。消费者是否可以触发 GC 可以继续讨论；关键约束是 GC 必须基于全局 root set 判断，不能因为某个用户触发 GC 就影响其他用户仍在保活的对象。

正常维护模型中不应存在“直接删除某个 object”的操作。对象生命周期由全量意图形成的 root set 和 GC 决定：用户通过修改自己的全量意图放弃保活，daemon 根据全局 root set 判断哪些对象可以被回收。

在安全边界上，`imst daemon` 是 installed 内容的唯一供应方。共享 store 中的安装结果只能由 daemon 写入和发布；普通用户、上层包管理器和其他消费者不能绕过 daemon 直接创建或修改 installed store object。

### 10.2 用户意图

普通用户通过自己持有的真实配置表达诉求：

- 该配置表达单个用户的全量意图，也就是该用户当前希望 `imst` 保证存在并保活的完整集合。
- 该配置是用户自身持有的资产，可以由该用户完全控制。
- 对 `imst` 来说，这个文件由用户手写，还是由 `mise` 这样的上层工具代为维护，属于同一个安全边界内的事情；`imst` 不需要区分。

这里的关键是：用户表达“我需要什么”，但不直接写入 `installed` 目录。

### 10.3 信任连接

用户需要先和 `imst` 建立一次性信任连接。连接建立后，用户只需要修改自己持有的真实配置，就可以驱动 `imst` 工作，不需要每次安装或每次变更都重新授权。

这个逻辑联系具体通过什么机制建立，属于后续设计问题。可选方向包括：

- 通过用户组成员关系表达用户被允许接入。
- 通过固定位置的用户配置表达该用户的全量意图。
- 通过注册记录或发现机制建立 daemon 和用户配置之间的长期联系。

无论采用哪种实现，安全保证都需要满足这些概念约束：

- 用户必须处在被授权的消费者集合中。
- daemon 只能读取已建立信任连接用户的固定意图入口。
- daemon 不接受任意路径作为用户请求来源。
- 用户意图来源必须处在该用户的安全边界内。

这个约束让普通用户可以持续表达诉求，同时避免把共享 store 的请求入口变成任意文件读取或任意路径授权机制。

### 10.4 全量意图与 GC Root

因为用户意图是单个用户的全量意图，所以它不应被理解为追加式任务队列。daemon 读取它时，应把它视为该用户当前 desired state 的快照：其中出现的对象代表该用户希望安装和保活；其中消失的对象代表该用户不再通过这份意图声明保活。

GC root 不是某一个用户的 `requests.json`，而是所有已建立信任连接用户的 `requests.json` 的集合。一个 store object 只要仍出现在任意用户的全量意图中，就应被视为仍需保活。最终是否可以 GC，要基于整个 root set，并结合管理员策略判断。

### 10.5 安装结果

daemon 根据用户全量意图安装包内容：

- `name` 来自 package spec 的 `spec.name`。
- `rev` 来自 `hash(spec)`，必须由确定性算法计算得到。
- 安装结果由 `{name, rev}` 识别。
- `rev` 算法必须公开、稳定，使上层工具可以在 daemon 完成安装前预测安装结果身份。
- 安装完成后，结果应进入只读、可共享、可复用状态。
- 同一个用户全量意图中允许出现多个相同 `name` 的 package spec。只要 spec 不同，计算得到的 `rev` 就不同，最终会形成不同的安装结果。

整个安装过程在概念上可以看作一个纯函数：

```text
fn(spec) -> Installed
```

也就是说，package spec 是输入，Installed 是确定性结果。daemon 的职责是把这个结果物化到共享 store，并完成校验、处理、权限收敛和只读发布。daemon 运行过程会有下载、缓存、状态记录等工程副作用，但这些副作用不应改变 `fn(spec) -> Installed` 的概念模型。

具体路径布局、索引方式和元数据位置留给 RFC 讨论。

### 10.6 状态观察

`imst` 需要向上层工具和管理员表达状态：

- 某个 package spec 是否已被接受。
- 对应 store object 是否正在获取、校验、处理或已经 ready。
- 失败原因是否可诊断。
- 某个 ready store object 的 store path 是什么。

上层包管理器如何把这些状态呈现给普通用户，属于上层工具自己的体验设计，不属于 `imst` 的职责。

### 10.7 包表达

一个包大致由这些要素表达：

```json
{
  "name": "example",
  "items": [
    {
      "src": "https://example.invalid/example.tar.gz",
      "digest": "sha256:...",
      "kind": "archive"
    }
  ]
}
```

其中：

- `name` 表示 package spec 的名称，也就是 `spec.name`。
- `items` 表示构成该包的一个或多个输入。
- `src` 表示输入来源。
- `digest` 表示输入内容校验。
- `kind` 表示输入如何进入 store object。

候选 `kind` 包括归档、单个可执行文件和普通文件。归档类输入可以有有限的声明式选项，例如选择展开方式或规范化行为；这些选项不能扩展成任意脚本能力。

`requests.json` 可以包含多个 package spec。每个 package spec 都有自己的 `name`，安装完成后对应一个独立的安装结果。一个 package spec 可以包含一个或多个 `items`，这些 `items` 都会被安装到该 package spec 对应的安装结果中。只有该 package spec 的所有 `items` 都完成获取、校验和处理以后，整个 package spec 才被认为 ready，并作为一个整体一次性暴露出来。

也就是说，`items` 是单个 package spec 的输入集合，安装结果是这些输入经过确定性处理后的 store object。`rev` 来自 `hash(spec)`，因此同名但不同 spec 的包可以同时存在。例如同一个 `name` 因版本不同而拥有不同下载地址或 digest 时，会得到不同的 `rev`，并形成不同的安装结果。这也是安装结果身份需要包含 `rev` 的原因。

`rev` 的计算算法必须是 public knowledge，而不是 daemon 内部私有细节。使用方可以根据同一份 package spec 自行计算 `rev`，从而在提交意图、观察状态或预测 store path 时使用同一个身份。

`items` 不是依赖列表，也不表达包之间的依赖图。它只表示构成单个安装结果所需的输入集合。依赖解析、版本选择和依赖图解释都属于上层包管理器。

`requests.json` 中的 spec 不是任意执行计划，而是对有限 store 能力的声明。它可以表达从某个 `src` 获取内容、用 `digest` 校验内容、按 `kind` 进行受限处理，但不应表达执行脚本或任意命令。与此同时，spec 中也存在一些要求和假设并不一定能被 `imst` 完全校验，例如 `src` 支持的源头应满足 immutable 假设：同一地址应获得确定性内容。`imst` 可以校验本次获取的内容是否匹配 digest，但不一定能证明该地址在所有时间、所有观察者下都不可变。

## 11. 开放问题

- 这个项目的核心问题是什么？
- 项目名称 `imst` 是否最终确定？
- `hash(spec)` 暂定使用 `sha256(canonical_json(spec))`；长期 canonical encoding 和版本化策略应如何定义？
- 多用户共享 store 的权限模型是什么？
- store object 从供应状态进入只读发布状态时，需要满足哪些权限收敛语义？
- 只读控制需要达到什么安全语义，才能支撑多用户共享？
- 如何调和使用方的用户态按需安装诉求，和供应方写入共享 store 所需的特权诉求？
- 特权边界应该覆盖哪些能力：获取、校验、处理、发布、状态维护、GC？
- 如果包内容本身包含可写目录、异常权限位或不适合共享的文件属性，`imst` 应拒绝、规范化，还是记录并交给上层？
- `src` 支持哪些源头类型，才能满足 immutable source 的假设？
- 当 `src` 不是严格 immutable，但提供 digest 可校验时，`imst` 是否接受？
- 有限能力集合的边界是什么，哪些声明式处理能力会越过边界？
- GC 如何避免误删仍被某个用户环境间接使用的对象？
- 除所有用户的 `requests.json` 之外，是否还需要管理员级 root 或系统级 pin？
- 保活粒度应该是什么：用户全量意图、工具版本、store object，还是某种上层 profile？
- 如果某个用户离开系统、配置被删除或 `mise` 状态损坏，保活声明如何过期或清理？
- GC 需要提供哪些可观察性和审计能力，才能让管理员信任？
- 有限初始化行为的边界是什么？允许 patch shebang、生成元数据、规范化权限吗？
- `imst` 面向上层工具应暴露怎样的能力边界？
- 信任连接如何证明用户属于被授权的消费者集合？
- 信任连接如何确保 daemon 观察的是该用户安全边界内的真实意图来源？
- 首次注册如何形成一次性授权，使 `imst daemon` 后续可以持续观察该用户请求而无需每次重新授权？
- 注册后的逻辑联系是否允许变更？如果允许，谁可以变更，如何审计？
- `imst` 需要向上层工具和管理员暴露哪些状态语义？
- 安装结果的外部身份是 `{name, rev}`，还是还需要额外的内容身份？
- 项目是否要对标 Nix store、OCI image layer/cache、Bazel repository cache、Maven/npm cache，或完全避开这些定位？
